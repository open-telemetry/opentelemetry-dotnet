# OpenTelemetry .NET Diagnostic: OTEL1006

## Overview

This is an Experimental API diagnostic covering the following APIs:

* `MetricStreamConfiguration.EnableLazyAllocation.get`
* `MetricStreamConfiguration.EnableLazyAllocation.set`
* `MeterProviderBuilderExtensions.SetDefaultLazyAllocation`

Experimental APIs may be changed or removed in the future.

## Details

By default the Metrics SDK eagerly allocates storage for `CardinalityLimit + 2`
metric points per metric stream as soon as the stream is created. For workloads
that configure a high `CardinalityLimit` to bound worst-case cardinality but
only ever observe a small number of time series, this up-front allocation can
dominate the SDK's managed-heap footprint.

These APIs add an opt-in lazy allocation mode. When lazy allocation is enabled,
metric point storage is allocated in fixed-size chunks on demand as new time
series are observed, up to the configured `CardinalityLimit`. The
spec-mandated reserved slots (no-tags and overflow) and the overflow behavior
at `CardinalityLimit` are preserved.

The MeterProvider default can be set with
`MeterProviderBuilderExtensions.SetDefaultLazyAllocation`, and any individual
metric stream can override that default with
`MetricStreamConfiguration.EnableLazyAllocation`:

```csharp
using var meterProvider = Sdk.CreateMeterProviderBuilder()
    .AddMeter("MyCompany.MyProduct")

    // Enable lazy allocation by default for every metric stream created by
    // this provider.
    .SetDefaultLazyAllocation(true)

    // A view can opt back out (or opt in) for a specific stream.
    .AddView(
        "my-low-cardinality-counter",
        new MetricStreamConfiguration
        {
            CardinalityLimit = 10,
            EnableLazyAllocation = false,
        })

    .AddView(
        "my-high-cardinality-counter",
        new MetricStreamConfiguration
        {
            CardinalityLimit = 100_000,
            EnableLazyAllocation = true,
        })

    .Build();
```

`EnableLazyAllocation` is nullable: when `null` (the default) the
MeterProvider-level default is used; when explicitly set to `true` or `false`
the view value wins over the MeterProvider default for that metric stream.

The default behavior of the SDK is unchanged: callers who do not opt in
continue to get eager allocation.

We are exposing these APIs experimentally for the following reasons:

* The lazy storage path uses a different internal data structure
  (a segmented metric point store) than the eager path. We want feedback from
  users who try it on real workloads before committing to the data structure
  and growth policy as a stable contract.

* The on/off shape of the API (`bool` on the MeterProvider and `bool?` on the
  view) is the simplest opt-in we could surface; we may want to evolve it
  (for example, to expose initial/chunk-size knobs, or to make lazy allocation
  the default once we are confident the trade-offs are right for all
  workloads).

**TL;DR** We want to gather feedback on the usability of the API, the
performance characteristics on real-world workloads, and whether the current
configuration shape is sufficient before exposing a stable API.
